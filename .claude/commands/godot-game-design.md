# 命令参数说明

- **关键词/主题** (必需): 游戏的核心概念、主题或设计方向，如“太空探索”“魔法学院”“赛博朋克”“解谜逃脱”等
- **游戏类型** (必需): 具体的游戏类型，如动作、平台跳跃、RPG、解谜、策略、冒险等
- **目标平台** (可选): 目标发布平台，如PC、移动端、Web、主机等，默认为PC
- **项目规模** (可选): 项目规模预期，如小型、中型、大型，默认为中型

# 执行步骤

## 第一阶段：概念分析与创意生成
### 工具使用：tavily-mcp + sequential-thinking + memory

1. **关键词解析与分析** (使用 sequential-thinking 工具)
   - 深度分析关键词的游戏设计潜力
   - 识别核心创意元素和设计约束
   - 评估概念的技术实现可行性

2. **市场调研与趋势分析** (使用 tavily-mcp 工具)
   - 搜索同类游戏的最新设计趋势
   - 分析竞品的核心玩法和独特卖点
   - 收集玩家反馈和市场需求数据

3. **创意发散与概念生成** (使用 memory 工具)
   - 基于关键词生成3-5个不同游戏概念
   - 查询记忆中的成功游戏设计模式
   - 设计核心玩法创新点

4. **概念筛选与评估** (使用 sequential-thinking 工具)
   - 评估各概念的创新性和市场潜力
   - 分析技术实现难度和资源需求
   - 筛选出最具潜力的1-2个概念

## 第二阶段：核心机制与系统设计
### 工具使用：sequential-thinking + memory

1. **核心循环设计**
   - 设计游戏的核心玩法循环
   - 定义玩家行为与系统反馈的关系
   - 确保循环的平衡性和可重复性

2. **游戏系统设计**
   - 设计支撑核心玩法的各项系统
   - 定义系统间的交互关系
   - 考虑扩展性和模块化设计

3. **平衡性与难度曲线**
   - 设计游戏的难度递进机制
   - 平衡奖励与挑战的关系
   - 考虑不同技能水平玩家的体验

4. **交互设计优化**
   - 设计玩家与游戏的交互方式
   - 确保控制方案的直观性
   - 设计反馈系统的及时性

## 第三阶段：完整文档生成
### 工具使用：filesystem + memory

1. **游戏概述文档编写**
   - 编写游戏的高层次描述
   - 定义目标受众和市场定位
   - 提炼核心卖点和竞争优势

2. **详细机制设计文档**
   - 详细阐述游戏机制和系统
   - 提供具体的数值设计和平衡参数
   - 包含游戏流程和状态转换图

3. **用户体验设计规范**
   - 设计UI/UX布局和交互流程
   - 定义反馈系统（视觉、听觉、触觉）
   - 制定新手引导和玩家成长路径

4. **技术实现建议文档**
   - 提供Godot实现的具体技术建议
   - 推荐合适的节点结构和架构模式
   - 包含性能优化和扩展性考虑

# Agent 与 MCP 工具指定

## 主要执行 Agent

### godot-game-designer (主导 Agent)
- **使用阶段**: 全流程主导
- **核心职责**: 创意游戏设计概念生成、完整GDD文档编写
- **专业能力**: 游戏创意构思、核心机制设计、用户体验优化、GDD标准化文档
- **主要工具**: 
  - `sequential-thinking`: 复杂概念分析和设计决策
  - `memory`: 存储和查询游戏设计知识库
  - `tavily-mcp`: 市场调研和趋势分析

### godot-architect (技术支持 Agent)
- **使用阶段**: 第二阶段后期和第三阶段
- **触发条件**: 需要具体Godot技术实现建议时
- **核心职责**: Godot技术架构设计、性能优化建议
- **专业能力**: 场景结构设计、SOLID原则应用、项目架构优化
- **主要工具**: 
  - `sequential-thinking`: 架构设计决策分析
  - `memory`: 查询Godot最佳实践和设计模式

### design-doc-writer (文档整理 Agent)
- **使用阶段**: 第三阶段文档整理
- **触发条件**: 需要标准化格式输出时
- **核心职责**: 文档结构化、格式标准化、技术文档撰写
- **专业能力**: 技术文档组织、标准化格式应用、内容结构优化
- **主要工具**: 
  - `filesystem`: 文件读写和格式处理
  - `memory`: 文档模板和标准格式查询

## MCP 工具使用策略

### 1. tavily-mcp 工具
- **使用时机**: 第一阶段市场调研
- **具体用途**: 
  - 搜索最新游戏设计趋势
  - 分析竞品游戏特点
  - 获取玩家反馈和市场数据
- **搜索策略**: 使用关键词+游戏类型+最新时间范围

- **使用时机**: 概念分析、设计决策、问题解决
- **具体用途**:
  - 深度分析关键词的游戏设计潜力
  - 评估不同概念的创新性和可行性
  - 解决设计过程中的复杂问题
- **思考深度**: 根据问题复杂度调整思考步骤数量

### 3. memory 工具
- **使用时机**: 全流程知识支持
- **具体用途**:
  - 存储游戏设计知识和最佳实践
  - 查询成功游戏的机制设计
  - 记录项目特定的设计约束和要求
- **记忆策略**: 为每个项目建立独立的记忆实体

### 4. filesystem 工具
- **使用时机**: 第三阶段文档输出
- **具体用途**:
  - 创建标准化的GDD文档文件
  - 生成markdown格式的设计文档
  - 保存设计过程中的重要决策记录
- **文件组织**: 按项目和阶段组织文档结构

# 交付件说明

## 主要交付物清单

### 1. 游戏概念文档 (Game Concept Document)
**文件名格式**: `[项目名称]_Concept_[日期].md`

**包含内容**:
- 游戏名称、类型和主题
- 核心玩法概述（1-2句话描述）
- 目标受众分析和市场定位
- 独特卖点(USP)和竞争优势
- 初步美术风格和音效设计方向
- 技术和资源需求初步评估

**文档结构**:
```markdown
# [游戏名称] - 概念文档

## 1. 游戏概述
## 2. 核心玩法
## 3. 目标受众
## 4. 独特卖点
## 5. 初步技术评估
```

### 2. 完整游戏设计文档 (GDD)
**文件名格式**: `[项目名称]_GDD_[版本]_[日期].md`

**包含内容**:
- **游戏概述**: 详细的游戏描述和背景故事
- **核心机制**: 完整的玩法循环和机制设计
- **游戏系统**: 进程系统、经济系统、战斗系统等
- **用户体验**: UI/UX设计、反馈系统、新手引导
- **内容设计**: 关卡设计框架、角色设计、道具系统
- **技术规格**: Godot实现建议、架构设计、性能考虑

**文档结构**:
```markdown
# [游戏名称] - 游戏设计文档 (GDD)

## 1. 游戏概述
### 1.1 游戏简介
### 1.2 目标受众
### 1.3 独特卖点

## 2. 核心机制设计
### 2.1 核心玩法循环
### 2.2 控制方案
### 2.3 核心交互机制

## 3. 游戏系统设计
### 3.1 进程系统
### 3.2 经济系统
### 3.3 战斗/挑战系统

## 4. 用户体验设计
### 4.1 UI/UX设计
### 4.2 反馈系统
### 4.3 新手引导

## 5. 内容设计框架
### 5.1 关卡设计原则
### 5.2 角色/实体设计
### 5.3 道具和奖励系统

## 6. 技术实现建议
### 6.1 Godot架构建议
### 6.2 关键系统实现
### 6.3 性能优化建议
```

### 3. 技术实现指南 (Technical Implementation Guide)
**文件名格式**: `[项目名称]_TechGuide_[日期].md`

**包含内容**:
- **Godot项目结构建议**: 推荐的文件夹组织和命名规范
- **节点架构设计**: 主要场景的节点树结构建议
- **关键脚本设计**: 核心系统的脚本设计模式
- **性能优化方案**: 基于目标平台的优化策略
- **开发里程碑**: 建议的开发阶段和里程碑规划

### 4. 设计决策记录 (Design Decision Log)
**文件名格式**: `[项目名称]_Decisions_[日期].md`

**包含内容**:
- 重要设计决策的记录和理由
- 备选方案的比较和选择依据
- 设计约束和限制条件
- 后续验证和测试计划

## 文档格式标准

### 文件格式
- **主要格式**: Markdown (.md)
- **编码**: UTF-8
- **行结尾**: LF (Unix风格)
- **文件命名**: 使用描述性名称，包含日期和版本信息

### 文档结构
- **层次化**: 使用标准的markdown标题层次 (H1-H6)
- **模块化**: 将大型文档拆分为逻辑清晰的模块
- **交叉引用**: 使用markdown链接进行文档间的交叉引用
- **图表支持**: 包含mermaid图表、表格和代码示例

### 版本控制
- **版本管理**: 每个文档包含版本号和更新日期
- **变更记录**: 文档末尾包含变更历史记录
- **备份策略**: 重要设计决策有独立的备份记录

# 约束与限制条件

## 设计约束条件

### 创新性约束
- **独特性要求**: 游戏概念必须具有明显的创新点或差异化特征
- **市场差异化**: 与现有同类游戏相比，至少有2-3个显著的差异化特点
- **玩法创新**: 核心机制设计避免完全复制现有游戏的成功模式

### 可行性约束
- **技术实现难度**: 所有设计建议必须考虑小团队或独立开发者的实现能力
- **资源限制**: 设计方案必须考虑合理的开发时间和人力成本
- **Godot引擎限制**: 所有技术建议必须在Godot引擎的能力范围内
- **平台兼容性**: 设计必须适配目标平台的硬件和软件限制

### 完整性约束
- **文档详细度**: 提供足够详细的设计信息，能够指导开发团队进行实际开发
- **可执行性**: 所有设计建议必须是可测试和可验证的
- **一致性**: 确保文档内部的逻辑一致性和技术可行性

## 技术约束条件

### Godot引擎约束
- **版本兼容性**: 设计建议基于当前稳定版本的Godot引擎
- **性能要求**: 设计的游戏性能目标必须符合目标设备的能力
- **平台适配**: 考虑不同平台（PC、移动端、Web）的技术差异
- **内存管理**: 设计必须考虑合理的内存使用和资源管理策略

### 开发约束
- **团队规模**: 设计方案考虑小型团队（1-10人）的开发能力
- **开发周期**: 建议的开发周期符合独立游戏的实际情况
- **技能要求**: 技术实现建议的技能要求在独立开发者可达范围内
- **工具依赖**: 尽量使用Godot内置功能，减少对外部工具的依赖

## 质量约束条件

### 游戏性约束
- **可玩性优先**: 所有机制设计必须以提升游戏体验为核心目标
- **平衡性要求**: 确保游戏的挑战性和可获得性的平衡
- **重玩价值**: 设计应考虑游戏的重玩价值和长期吸引力

### 用户体验约束
- **易学性**: 游戏的核心玩法应该在5-10分钟内被理解
- **可预测性**: 游戏规则和反馈系统应该符合玩家预期
- **容错性**: 设计应该允许玩家犯错并提供恢复机制

### 扩展性约束
- **内容扩展**: 核心系统设计应支持后续内容的扩展
- **平台扩展**: 设计考虑未来可能的平台移植需求
- **功能扩展**: 预留关键功能的扩展接口和升级路径

## 合规性约束

### 法律合规
- **版权要求**: 所有游戏概念和设计必须避免侵犯现有知识产权
- **平台规范**: 符合目标平台的发布规范和内容政策
- **数据隐私**: 如涉及用户数据，必须符合相关隐私保护法规

### 伦理约束
- **内容适宜性**: 游戏内容符合目标受众的年龄和文化背景
- **社会责任**: 避免包含可能引起争议或不良影响的内容
- **可访问性**: 考虑不同能力玩家的可访问性需求

# 使用场景与最佳实践

## 适用场景

### 新项目启动阶段
- **完整概念设计**: 从零开始创建全新的游戏概念
- **项目立项支持**: 为项目立项提供完整的设计文档
- **团队协作基础**: 为开发团队提供清晰的设计方向和目标
- **投资演示材料**: 制作用于投资或合作演示的设计文档

### 创意构思阶段
- **灵感触发**: 基于关键词生成游戏设计灵感
- **创意验证**: 评估创意概念的可行性和市场潜力
- **多方案对比**: 为同一主题生成多个不同的设计方案
- **创新探索**: 探索特定类型游戏的创新可能性

### 设计深化阶段
- **机制细化**: 将初步想法深化为详细的游戏机制
- **系统设计**: 设计复杂的游戏系统和交互关系
- **平衡性设计**: 分析和优化游戏的平衡性和难度曲线
- **用户体验优化**: 完善UI/UX设计和玩家体验流程

### 文档标准化阶段
- **GDD创建**: 生成符合行业标准的设计文档
- **团队沟通**: 创建便于团队成员理解和执行的文档
- **开发指导**: 为实际开发提供详细的技术指导
- **项目管理**: 建立项目进度跟踪和里程碑管理基础

## 不适用场景

### 技术实现场景
- **具体代码编写**: 需要使用 `godot-game-developer` 命令
- **性能调试优化**: 需要使用 `godot-code-reviewer` 命令进行代码审查
- **架构重构**: 需要使用 `godot-architect` 命令进行深度架构设计
- **bug修复**: 需要使用 `godot-game-developer` 命令进行技术问题解决

### 非设计相关工作
- **市场推广**: 营销策略和推广方案设计
- **商业运营**: 商业模式和运营策略规划
- **美术制作**: 具体的美术资源和素材制作
- **音效设计**: 具体的音乐和音效制作

## 最佳实践指南

### 输入参数优化

#### 关键词/主题设计
- **具体明确**: 使用具体的概念而非抽象描述，如“太空探索救援”而非“太空游戏”
- **组合创新**: 结合2-3个不同元素创造独特性，如“时间操控+平台跳跃+解谜”
- **情感定位**: 包含期望的情感体验，如“紧张刺激”“放松治愈”“史诗感”
- **避免冲突**: 确保关键词之间没有本质的冲突，如“写实风格+卡通角色”

#### 游戏类型选择
- **明确分类**: 使用标准的游戏类型分类，避免自创类型名称
- **混合类型**: 如需要混合类型，明确主次关系，如“动作+RPG”
- **平台适配**: 考虑目标平台适合的游戏类型特点
- **市场需求**: 结合当前市场需求和趋势选择类型

#### 平台和规模设定
- **现实评估**: 根据团队规模和技能水平设定合理的目标平台
- **技术约束**: 考虑不同平台的技术限制和优化需求
- **用户习惯**: 考虑目标平台用户的使用习惯和期望

### 输出文档使用建议

#### 文档管理
- **版本控制**: 使用Git等工具管理设计文档的版本迭代
- **定期更新**: 根据开发进展和市场变化及时更新设计文档
- **团队同步**: 确保团队成员都使用最新版本的设计文档
- **变更记录**: 记录重要的设计变更和决策理由

#### 开发执行
- **分阶段验证**: 在开发过程中分阶段验证设计假设
- **原型测试**: 使用简单原型测试核心机制的可行性
- **用户反馈**: 定期收集目标用户群体的反馈意见
- **灵活调整**: 根据实际情况灵活调整设计方案

#### 质量保证
- **设计审查**: 定期组织设计审查会议，评估设计质量
- **竞品分析**: 持续关注竞品动态，调整差异化策略
- **技术预研**: 对关键技术点进行前瞻性研究
- **风险控制**: 识别设计风险并制定应对方案

# 命令执行限制说明

- **代码实现**: 使用 `godot-game-developer` 命令
- **代码审查**: 使用 `godot-code-reviewer` 命令
- **架构设计**: 使用 `godot-architect` 命令
- **性能优化**: 使用 `godot-code-reviewer` 命令
