# 参数说明

- **关键词/主题** (必需): 游戏的核心概念、主题或设计方向，如"太空探索""魔法学院""赛博朋克""解谜逃脱"等
- **游戏类型** (必需): 具体的游戏类型，如动作、平台跳跃、RPG、解谜、策略、冒险等
- **目标平台** (可选): 目标发布平台，如PC、移动端、Web、主机等，默认为PC
- **项目规模** (可选): 项目规模预期，如小型、中型、大型，默认为中型

# 执行步骤

**当前游戏设计目标**: $ARGUMENTS

## 第一阶段：概念分析与创意生成
工具：tavily-mcp + sequential-thinking + memory

1. **关键词解析与分析**
   - 深度分析关键词的游戏设计潜力
   - 识别核心创意元素和设计约束
   - 评估概念的技术实现可行性

2. **市场调研与趋势分析**
   - 搜索同类游戏的最新设计趋势
   - 分析竞品的核心玩法和独特卖点
   - 收集玩家反馈和市场需求数据

3. **创意发散与概念生成**
   - 基于关键词生成3-5个不同游戏概念
   - 查询记忆中的成功游戏设计模式
   - 设计核心玩法创新点

4. **概念筛选与评估**
   - 评估各概念的创新性和市场潜力
   - 分析技术实现难度和资源需求
   - 筛选出最具潜力的1-2个概念

## 第二阶段：核心机制与系统设计
工具：sequential-thinking + memory

1. **核心循环设计**
   - 设计游戏的核心玩法循环
   - 定义玩家行为与系统反馈的关系
   - 确保循环的平衡性和可重复性

2. **游戏系统设计**
   - 设计支撑核心玩法的各项系统
   - 定义系统间的交互关系
   - 考虑扩展性和模块化设计

3. **平衡性与难度曲线**
   - 设计游戏的难度递进机制
   - 平衡奖励与挑战的关系
   - 考虑不同技能水平玩家的体验

4. **交互设计优化**
   - 设计玩家与游戏的交互方式
   - 确保控制方案的直观性
   - 设计反馈系统的及时性

## 第三阶段：完整文档生成
工具：filesystem + memory

1. **游戏概述文档编写**
   - 编写游戏的高层次描述
   - 定义目标受众和市场定位
   - 提炼核心卖点和竞争优势

2. **详细机制设计文档**
   - 详细阐述游戏机制和系统
   - 提供具体的数值设计和平衡参数
   - 包含游戏流程和状态转换图

3. **用户体验设计规范**
   - 设计UI/UX布局和交互流程
   - 定义反馈系统（视觉、听觉、触觉）
   - 制定新手引导和玩家成长路径

4. **技术实现建议文档**
   - 提供Godot实现的具体技术建议
   - 推荐合适的节点结构和架构模式
   - 包含性能优化和扩展性考虑

# Agent 与 MCP 工具

## 主要 Agent
- **godot-game-designer** (主导): 创意游戏设计概念生成、完整GDD文档编写
- **godot-architect** (技术支持): Godot技术架构设计、性能优化建议
- **design-doc-writer** (文档整理): 文档结构化、格式标准化、技术文档撰写

## MCP 工具
- **tavily-mcp**: 市场调研、趋势分析、竞品研究
- **sequential-thinking**: 概念分析、设计决策、问题解决
- **memory**: 游戏设计知识库、最佳实践查询、项目信息存储
- **filesystem**: 文档创建、格式处理、文件组织

# 交付件

## 主要文档
1. **游戏概念文档**: 游戏名称、类型、核心玩法、目标受众、独特卖点、初步技术评估
2. **完整游戏设计文档(GDD)**: 游戏概述、核心机制、游戏系统、用户体验、内容设计、技术规格
3. **技术实现指南**: Godot项目结构建议、节点架构设计、关键脚本设计、性能优化方案
4. **设计决策记录**: 重要设计决策记录、备选方案比较、设计约束、验证计划

# 约束条件

## 设计约束
- **创新性要求**: 游戏概念必须具有明显的创新点或差异化特征
- **技术可行性**: 所有设计建议必须考虑小团队或独立开发者的实现能力
- **资源限制**: 设计方案必须考虑合理的开发时间和人力成本
- **Godot引擎限制**: 所有技术建议必须在Godot引擎的能力范围内

## 技术约束
- **版本兼容性**: 设计建议基于当前稳定版本的Godot引擎
- **性能要求**: 设计的游戏性能目标必须符合目标设备的能力
- **平台适配**: 考虑不同平台（PC、移动端、Web）的技术差异
- **团队规模**: 设计方案考虑小型团队（1-10人）的开发能力

## 质量约束
- **可玩性优先**: 所有机制设计必须以提升游戏体验为核心目标
- **平衡性要求**: 确保游戏的挑战性和可获得性的平衡
- **易学性**: 游戏的核心玩法应该在5-10分钟内被理解
- **扩展性**: 核心系统设计应支持后续内容的扩展

# 使用场景

## 适用场景
- **新项目启动**: 从零开始创建全新的游戏概念、项目立项支持、团队协作基础
- **创意构思**: 基于关键词生成游戏设计灵感、创意验证、多方案对比、创新探索
- **设计深化**: 机制细化、系统设计、平衡性设计、用户体验优化
- **文档标准化**: GDD创建、团队沟通、开发指导、项目管理

## 不适用场景
- **代码实现**: 使用 `godot-game-developer` 命令
- **代码审查**: 使用 `godot-code-reviewer` 命令
- **架构设计**: 使用 `godot-architect` 命令
- **问题调试**: 使用 `godot-debug` 命令
