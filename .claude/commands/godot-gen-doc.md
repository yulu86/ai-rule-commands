# 参数说明

- **主题/特性** (必需): 文档生成主题，如"战斗系统"、"存档系统"、"玩家控制"等
- **详细程度** (可选): 概要模式(高层次概述)或详细模式(完整设计文档)，默认详细模式

# 执行步骤

**当前目标**: $ARGUMENTS

## 第一阶段：需求分析与资料收集
工具：sequential-thinking + memory + tavily-mcp

1. 主题解析与范围界定
2. 知识收集与研究
3. 设计标准确立

## 第二阶段：内容规划与架构设计
工具：design-doc-writer + sequential-thinking

1. 文档架构设计
2. 内容深度规划
3. 技术方案设计

## 第三阶段：文档生成与内容创建
工具：design-doc-writer + godot-architect + memory

1. 核心内容生成
2. 技术架构文档
3. 实践指导完善

## 第四阶段：质量保证与优化
工具：design-doc-writer + sequential-thinking

1. 文档质量审查
2. 用户体验优化
3. 完整性验证

# Agent 与 MCP 工具

## 主要 Agent
- **design-doc-writer** (主导): 文档结构设计、内容组织、技术文档撰写
- **godot-architect** (技术支持): 技术架构设计、系统方案规划
- **godot-game-designer** (游戏设计支持): 游戏机制设计、用户体验规划

## MCP 工具
- **sequential-thinking**: 需求分析、架构决策、质量优化
- **memory**: 知识查询、最佳实践、模板应用
- **tavily-mcp**: 趋势研究、案例收集
- **filesystem**: 文档创建、编辑、格式处理

# 交付件

## 主要文档
1. **游戏设计文档(GDD)**: 完整的游戏概念和机制设计
2. **技术架构文档**: 详细的系统架构和设计方案
3. **开发指导文档**: 实施指导、最佳实践、配置指南

## 文档结构
```markdown
# 游戏设计文档：[主题]

## 1. 游戏概述
### 1.1 游戏概念和定位
### 1.2 核心玩法设计
### 1.3 目标受众分析

## 2. 核心机制设计
### 2.1 基础玩法机制
### 2.2 核心循环设计
### 2.3 规则和条件设定

## 3. 系统功能设计
### 3.1 进程系统
### 3.2 奖励机制
### 3.3 社交功能

## 4. 用户体验设计
### 4.1 UI/UX设计原则
### 4.2 交互流程设计
### 4.3 反馈系统设计
```

# 约束条件

## 设计约束
- **专业性**: 确保技术内容的准确性和可行性
- **完整性**: 覆盖主题的所有重要方面
- **创新性**: 包含原创的设计思路和技术方案

## 技术约束
- **Godot版本**: 基于指定Godot版本进行设计
- **实现难度**: 考虑小团队或独立开发者的实现能力
- **性能要求**: 考虑目标平台的性能限制

# 使用场景

## 适用场景
- **新项目启动**: 完整项目的设计文档生成
- **功能规划**: 特定系统或功能的详细设计
- **团队协作**: 为开发团队提供设计指导和技术规范
- **项目交接**: 生成标准化的技术文档和知识库

## 不适用场景
- **具体代码实现**: 使用 `godot-gen-code` 命令
- **架构设计**: 使用 `godot-architect` 命令
- **代码审查**: 使用 `godot-code-reviewer` 命令
- **问题调试**: 使用 `godot-debug` 命令
