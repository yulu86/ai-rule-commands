# 命令参数说明

- **需求文档** (必需): 指定开发计划的基础文档，可以是：
  - **完整需求规格**: 详细的软件需求规格说明书(SRS)
  - **功能需求列表**: 系统功能需求清单和特性描述
  - **用户研究报告**: 用户画像、用户场景和用户需求分析
  - **市场需求数据**: 市场调研和竞品分析报告
  - **业务目标文档**: 业务目标和价值主张描述
  - **现有系统分析**: 现有系统的问题分析和改进需求

- **架构设计** (可选): 指定架构设计参考文档，用于技术实现指导
  - **系统架构文档**: 完整的系统架构设计(ADD)
  - **技术架构图**: 技术架构和组件关系图
  - **数据架构设计**: 数据模型和数据流设计
  - **接口设计规范**: API和接口设计标准
  - **部署架构**: 系统部署和基础设施架构
  - **默认值**: 如未指定，基于需求类型推荐架构参考

- **拆解深度** (可选): 指定User Story拆解的详细程度和粒度
  - **史诗级别**: 生成高层次的史诗用户故事(Epic)
  - **特性级别**: 生成中等粒度的功能特性故事
  - **故事级别**: 生成标准的独立用户故事
  - **任务级别**: 生成细粒度的具体开发任务
  - **默认值**: 如未指定，默认为故事级别

# 执行步骤

**当前开发计划目标**: $ARGUMENTS

## 第一阶段：需求理解与分析
### 工具使用：sequential-thinking + memory

1. **需求深度分析** (使用 sequential-thinking 工具)
   - 分析业务目标和用户价值主张
   - 识别核心业务流程和关键功能点
   - 理解用户角色和使用场景
   - 确定功能的优先级和依赖关系

2. **用户角色识别** (使用 sequential-thinking 工具)
   - 识别系统的目标用户群体
   - 定义用户画像和特征描述
   - 分析用户需求和痛点
   - 建立用户优先级和影响权重

3. **业务流程梳理** (使用 memory + sequential-thinking 工具)
   - 梳理核心业务流程和用户旅程
   - 识别关键决策点和交互点
   - 分析业务规则和约束条件
   - 确定流程优化和改进点

## 第二阶段：User Story设计
### 工具使用：sequential-thinking + memory

1. **Epic故事规划** (使用 sequential-thinking 工具)
   - 识别高层次的功能主题和业务价值
   - 设计史诗级用户故事和业务目标
   - 规划主要功能模块和特性范围
   - 制定Epic的验收标准和成功指标

2. **Feature故事拆解** (使用 sequential-thinking 工具)
   - 将Epic拆解为具体的功能特性
   - 设计特性级别的用户故事
   - 定义功能的边界和包含范围
   - 制定特性的验收条件和技术要求

3. **User Story创建** (使用 sequential-thinking 工具)
   - 按照标准User Story模板编写具体故事
   - 确保故事的独立性和可测试性
   - 定义清晰的验收标准(AC)
   - 估算故事的工作量和复杂度

## 第三阶段：技术实现规划
### 工具使用：software-architect + sequential-thinking

1. **架构适配分析** (使用 software-architect Agent)
   - 分析User Story与架构设计的匹配度
   - 评估技术实现的可行性和复杂度
   - 识别技术风险和关键依赖
   - 制定技术实施策略和优先级

2. **依赖关系分析** (使用 sequential-thinking 工具)
   - 分析User Story之间的技术依赖
   - 识别共享组件和公共资源
   - 制定开发顺序和迭代规划
   - 预防和解决潜在的冲突问题

3. **资源评估规划** (使用 sequential-thinking 工具)
   - 评估每个Story的开发工作量
   - 分析所需的技术能力和团队技能
   - 制定时间估算和资源分配
   - 规划团队协作和知识分享

## 第四阶段：文档生成与优化
### 工具使用：design-doc-writer + memory

1. **开发文档生成** (使用 design-doc-writer Agent)
   - 生成完整的User Story文档
   - 创建开发计划和迭代规划
   - 编写验收标准和测试用例
   - 制作交付清单和质量标准

2. **交付计划完善** (使用 memory 工具)
   - 完善开发里程碑和交付节点
   - 制定风险评估和应对策略
   - 创建沟通计划和协作流程
   - 补充相关的技术参考资料

3. **质量保证验证** (使用 sequential-thinking 工具)
   - 验证User Story的完整性和独立性
   - 确保验收标准的明确性和可测试性
   - 优化开发计划的可行性和效率
   - 建立持续改进和反馈机制

# User Story模板

## 标准User Story模板

### 基础格式
```
作为一个 [用户角色]
我希望 [功能描述]
以便 [业务价值/目标]

验收标准：
- [具体的可验证条件1]
- [具体的可验证条件2]
- [具体的可验证条件3]
```

### 完整格式
```
**User Story ID**: US_[模块缩写]_[序号]
**标题**: [简短明确的故事标题]
**用户角色**: [具体的用户角色]
**优先级**: [高/中/低]
**工作量**: [故事点/人天]

作为一个 [用户角色]
我希望 [功能描述]
以便 [业务价值/目标]

**验收标准 (Acceptance Criteria)**:
- [具体的可验证条件1]
- [具体的可验证条件2]
- [具体的可验证条件3]

**技术要求**:
- [实现的技术细节]
- [性能要求]
- [安全要求]
- [兼容性要求]

**依赖关系**:
- [前置依赖的User Story]
- [技术依赖的组件或服务]
- [数据依赖的接口或数据源]

**测试策略**:
- [单元测试要求]
- [集成测试要求]
- [端到端测试要求]
- [用户验收测试要求]

**交付标准**:
- [代码质量标准]
- [文档要求]
- [部署要求]
- [验收标准]
```

## User Story类型模板

### 功能型User Story
```
**类型**: 功能型
**模块**: [具体功能模块]

作为一个 [用户角色]
我希望 [具体的功能操作]
以便 [实现特定的业务目标]

**验收标准**:
- Given [前置条件]
  When [操作步骤]
  Then [预期结果]
- Given [另一前置条件]
  When [另一操作步骤]
  Then [另一预期结果]
```

### 非功能型User Story
```
**类型**: 非功能型
**关注点**: [性能/安全/可用性等]

作为一个 [系统角色]
我希望 [系统具备某种非功能特性]
以便 [提升用户体验或系统质量]

**验收标准**:
- [具体的性能指标]
- [安全合规要求]
- [可用性标准]
- [兼容性要求]
```

### 技术型User Story
```
**类型**: 技术型
**技术目标**: [重构/升级/优化等]

作为一个 [开发者/运维人员]
我希望 [技术改进或优化]
以便 [提升开发效率或系统质量]

**验收标准**:
- [技术改进的具体指标]
- [性能提升的测量标准]
- [代码质量的改进要求]
- [维护成本的降低目标]
```

# Agent 与 MCP 工具指定

## 主要执行 Agent

### software-architect (技术规划主导 Agent)
- **使用阶段**: 第三阶段技术实现规划、架构适配分析
- **核心职责**: 技术实现可行性分析、架构适配评估、技术风险管理
- **专业能力**: 软件架构设计、技术方案评估、开发流程优化
- **主要工具**:
  - `sequential-thinking`: 技术方案分析、依赖关系梳理
  - `memory`: 技术知识库查询、最佳实践应用
  - `filesystem`: 技术文档生成、架构图绘制

### design-doc-writer (文档生成主导 Agent)
- **使用阶段**: 第四阶段文档生成、交付计划完善
- **核心职责**: User Story文档撰写、开发计划制定、标准格式应用
- **专业能力**: 技术文档标准化、需求文档撰写、项目文档管理
- **主要工具**:
  - `filesystem`: 文档创建、模板应用、格式标准化
  - `sequential-thinking`: 文档结构设计、内容组织优化
  - `memory`: 文档模板查询、标准规范应用

### software-developer (开发实现支持 Agent)
- **使用阶段**: 第二阶段User Story设计、工作量评估
- **核心职责**: 开发可行性分析、工作量估算、技术实现指导
- **专业能力**: 软件开发实现、技术栈评估、开发流程优化
- **主要工具**:
  - `sequential-thinking`: 开发难度分析、技术方案评估
  - `memory`: 开发最佳实践、技术标准查询
  - `filesystem`: 开发指导文档、技术规范生成

### system-designer (系统设计支持 Agent)
- **使用阶段**: 第一阶段需求分析、用户角色识别
- **核心职责**: 用户体验设计、系统交互设计、业务流程优化
- **专业能力**: 用户体验设计、系统设计理论、交互设计原则
- **主要工具**:
  - `sequential-thinking`: 用户需求分析、业务流程梳理
  - `tavily-mcp`: 用户研究趋势、设计模式调研
  - `memory`: 设计原则应用、最佳实践查询

## MCP 工具使用策略

### 1. sequential-thinking 工具
- **使用时机**: 第一阶段需求分析、第二阶段User Story设计、第三阶段技术规划
- **具体用途**:
  - 深度分析业务需求和用户价值
  - 设计合理的User Story结构和拆解方案
  - 分析技术实现的依赖关系和风险
  - 制定可执行的开发计划和优先级
- **思考深度**: 根据拆解深度需求和项目复杂度调整分析深度

### 2. memory 工具
- **使用时机**: 第一阶段需求理解、第三阶段技术评估、全程知识支持
- **具体用途**:
  - 查询User Story设计模式和最佳实践
  - 收集相关的开发标准和技术规范
  - 存储重要的用户故事和开发决策
  - 建立项目主题的知识库和模板库
- **记忆策略**: 为每个项目建立独立的User Story知识图谱

### 3. tavily-mcp 工具
- **使用时机**: 第一阶段用户研究、第二阶段设计参考、第三阶段技术调研
- **具体用途**:
  - 搜索最新的用户体验设计和用户研究趋势
  - 收集同类产品的User Story案例和设计经验
  - 查询开发工具和技术栈的最新发展
  - 获取行业最佳实践和开发标准
- **搜索策略**: 使用业务领域+用户类型+功能场景进行搜索

### 4. filesystem 工具
- **使用时机**: 第四阶段文档生成、交付计划制定、全程文件管理
- **具体用途**:
  - 创建和编辑User Story文档
  - 组织开发计划和技术资料
  - 管理相关的调研文档和参考资料
  - 生成不同格式的开发输出文件
- **操作策略**: 遵循软件开发文档管理规范和版本控制要求

# 约束与限制条件

- 只专注于需求拆解和开发计划制定，不涉及具体代码实现
- User Story必须确保独立性和可测试性
- 所有交付物必须遵循标准的Story模板和格式

## Story设计约束

### 独立性约束
- **功能独立**: 每个User Story必须代表一个独立的功能价值
- **开发独立**: 故事之间应尽可能减少技术依赖
- **测试独立**: 每个故事都应有独立的验收标准和测试用例
- **交付独立**: 故事完成后应该能够独立交付和部署

### 可测试性约束
- **验收标准明确**: 每个故事必须有清晰可验证的验收标准
- **测试可执行**: 验收标准必须是可以通过测试验证的具体条件
- **结果可观察**: 故事的完成效果必须是用户可观察和验证的
- **指标可测量**: 故事的业务价值应该有可量化的评估指标

### 价值导向约束
- **用户价值**: 每个故事必须为特定用户角色提供明确价值
- **业务目标**: 故事应直接支持项目的业务目标和战略
- **优先级清晰**: 每个故事都应该有明确的业务优先级
- **价值可验证**: 故事的价值应该是可验证和可测量的

## 项目约束条件

### 范围约束
- **需求边界**: 严格遵循项目需求文档的范围和边界
- **功能完整性**: 确保拆解后的故事覆盖所有必要功能
- **排除冗余**: 避免创建重复或冗余的User Story
- **范围蔓延控制**: 防止在拆解过程中引入范围蔓延

### 质量约束
- **Story质量**: 确保每个User Story符合质量标准
- **验收标准完整性**: 验收标准必须完整和明确
- **技术可行性**: 所有故事必须在技术上可实现
- **时间合理性**: 工作量估算必须合理和可执行

### 资源约束
- **团队能力**: 考虑开发团队的技术能力和经验水平
- **时间限制**: 考虑项目的时间限制和交付压力
- **预算约束**: 评估开发成本和资源消耗
- **技能匹配**: 确保故事分配与团队能力相匹配

## 使用约束条件

### 文档质量约束
- **格式标准**: 严格遵循User Story的标准格式和模板
- **内容完整**: 确保故事信息的完整性和准确性
- **语言清晰**: 使用清晰、准确、无歧义的表达方式
- **版本控制**: 建立严格的版本控制和变更管理流程

### 验证责任约束
- **团队确认**: 重要User Story需要开发团队确认可行性
- **用户验证**: 关键功能故事需要最终用户或产品经理确认
- **技术评审**: 技术复杂的Story需要架构师或技术专家评审
- **持续更新**: 根据项目进展持续优化和完善User Story

## 伦理约束条件

### 专业责任约束
- **质量承诺**: 对拆解的User Story质量承担专业责任
- **价值导向**: 确保每个故事都为用户和业务创造真实价值
- **透明沟通**: 对故事的风险和复杂性保持透明沟通
- **持续改进**: 基于实施反馈持续改进拆解方法和模板

### 团队责任约束
- **协作促进**: 促进开发团队的协作和知识分享
- **技能发展**: 考虑团队成员的技能发展和学习机会
- **工作平衡**: 合理分配工作负载，避免团队过载
- **文化建设**: 促进积极的团队文化和工作氛围

# 使用场景与最佳实践

- **精益开发**: 创建最小可行产品(MVP)的用户故事
- **持续交付**: 支持持续交付的独立用户故事

- **功能扩展**: 规划新功能模块的用户故事集合
- **用户增长**: 设计支持用户增长的功能故事
- **产品迭代**: 基于用户反馈优化现有功能故事

### 系统重构项目
- **技术重构**: 将重构需求转化为技术型User Story
- **功能迁移**: 规划功能迁移和系统升级的故事
- **性能优化**: 拆解性能优化的具体实现故事

### 集成开发项目
- **系统集成**: 设计系统集成和数据对接的用户故事
- **API开发**: 拆解API开发的具体功能故事
- **第三方集成**: 规划第三方服务集成的实现故事
- **数据同步**: 创建数据同步和一致性保证的故事

### 项目管理
- **项目规划**: 不替代专业的项目管理和进度规划工具
- **资源管理**: 不进行详细的人力资源和成本管理
- **风险管理**: 不替代完整的项目风险管理体系
- **质量管理**: 不涉及全面的项目质量管理流程

### 技术架构
- **架构设计**: 不进行系统架构设计和技术选型
- **基础设施**: 不涉及基础设施规划和部署配置
- **技术栈选择**: 不进行技术栈的详细选择和评估
- **性能调优**: 不进行具体的系统性能优化和调优

### 业务咨询
- **业务战略**: 不提供业务战略和商业模式咨询
- **市场分析**: 不进行市场调研和竞品分析
- **用户研究**: 不替代专业的用户研究和需求收集
- **产品规划**: 不涉及长期产品路线图和战略规划

## 最佳实践指南

### User Story设计优化

#### INVEST原则应用
- **Independent**: 确保故事的独立性和无依赖性
- **Negotiable**: 保持故事的灵活性和可协商性
- **Valuable**: 确保每个故事都有明确的用户价值
- **Estimable**: 故事的工作量应该可以合理估算
- **Small**: 保持故事的小粒度和可完成性
- **Testable**: 故事必须具有明确的可测试性

#### 验收标准优化
- **具体明确**: 使用具体、可量化的验收条件
- **可观察性**: 验收结果应该是可观察和可验证的
- **完整性**: 覆盖所有重要的功能和场景
- **用户视角**: 从用户角度定义验收标准

#### 依赖关系管理
- **依赖最小化**: 尽量减少User Story之间的依赖关系
- **关键路径识别**: 识别项目关键路径上的重要依赖
- **并行规划**: 规划可以并行开发的Story集合
- **风险管理**: 提前识别和管理依赖风险

### 开发计划优化

#### 优先级排序
- **价值导向**: 基于业务价值和用户影响排序
- **风险评估**: 考虑技术风险和实施难度
- **依赖关系**: 考虑Story之间的依赖和约束
- **团队能力**: 考虑团队技能和可用资源

#### 迭代规划
- **平衡分配**: 在每个迭代中平衡不同类型的工作
- **容量规划**: 考虑团队的实际工作能力和效率
- **学习曲线**: 为新技术和复杂功能预留学习时间
- **风险缓解**: 在早期迭代中处理高风险的Story

#### 质量保证
- **定义清晰**: 确保每个Story都有清晰的质量标准
- **测试驱动**: 采用测试驱动的方法定义验收标准
- **持续集成**: 建立持续集成和自动化测试流程
- **用户参与**: 在整个开发过程中保持用户参与和反馈

### 团队协作优化

#### 沟通管理
- **Story共识**: 确保团队对每个Story有共同理解
- **定期评审**: 建立定期的Story评审和优化机制
- **知识分享**: 促进团队间的知识和经验分享
- **反馈收集**: 建立及时反馈和问题解决机制

#### 技能发展
- **技能匹配**: 根据团队能力合理分配Story
- **学习机会**: 为团队成员提供学习新技能的机会
- **导师制度**: 建立导师制度和知识传承机制
- **持续培训**: 提供持续的技术培训和技能提升
